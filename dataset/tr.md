# ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ТЗ)

**Название проекта:** Personal AI Memory Agent (Custom LLM)
**Цель:** Создание специализированной LLM (на базе Qwen 2.5), способной работать как автономный агент: вести естественный диалог, управлять долгосрочной памятью (Qdrant) и использовать поиск в интернете (Google) без галлюцинаций и лишних инструкций в коде.

---

## 1. Выбор Базовой Модели

* **Архитектура:** Qwen 2.5 (Instruct).
* **Размер:**
* **3B (рекомендуется):** Для высокой скорости работы на локальном ПК (CPU/слабая GPU).
* **7B:** Если есть видеокарта с 8GB+ VRAM, для более глубокого понимания контекста.


* **Квантование:** 4-bit (GGUF Q4_K_M) для инференса.

---

## 2. Функциональные Требования (Behavior)

Модель должна классифицировать входящие сообщения и выбирать одну из 4 стратегий поведения **автономно**, основываясь на контексте, а не на жестких правилах `if/else`.

### 2.1. Режимы работы

1. **Chitchat (Болтовня):** Поддержание разговора без использования инструментов.
2. **Memory Save (Сохранение):** Выявление фактов о пользователе и формирование команды сохранения.
3. **Intent Search (Поиск):** Определение потребности в информации (личной или внешней) и формирование поискового запроса.
4. **RAG Response (Ответ по данным):** Генерация финального ответа на основе предоставленных системой данных (`SYSTEM_OBSERVATION`).

### 2.2. Ограничения (Guardrails)

* **Запрет на филлеры:** Модель **никогда** не должна генерировать фразы: "Подождите", "Я ищу", "Секундочку", "Давайте посмотрим".
* **Запрет на галлюцинации ролей:** Модель не должна писать префиксы `User:`, `Assistant:`, `ОТВЕТ:`.
* **Формат мыслей:** Каждое действие должно предваряться блоком `МЫСЛЬ: [рассуждение]`.

---

## 3. Протокол Взаимодействия (Инструменты)

Модель должна строго следовать синтаксису команд.

### 3.1. Команда сохранения (SAVE)

* **Триггер:** Пользователь сообщает факт ("Я купил машину", "Не люблю лук").
* **Синтаксис:** `CMD:SAVE | Текст факта | тег1, тег2, тег3`
* **Требования к тегам:**
* Строчные буквы.
* Иерархия: `[Категория], [Сущность], [Детали]`.
* *Пример:* `CMD:SAVE | Аллергия на мед | здоровье, еда, аллергия`



### 3.2. Команда поиска в памяти (SEARCH)

* **Триггер:** Вопросы о прошлом или личности ("На чем я пишу?", "Какой код от двери?").
* **Синтаксис:** `CMD:SEARCH | поисковый запрос`
* **Логика:** Запрос должен быть перефразирован для лучшего поиска (User: "Что там с кодом?" -> Query: "код от двери домофона").

### 3.3. Команда поиска в Google (GOOGLE)

* **Триггер:** Вопросы о внешнем мире, новостях, погоде, курсах валют.
* **Синтаксис:** `CMD:GOOGLE | поисковый запрос`
* **Ограничение:** Не гуглить общеизвестные факты (история, наука, базовые определения).

---

## 4. Спецификация Датасета (Dataset)

Для обучения требуется сгенерировать синтетический датасет в формате **JSONL**.
**Объем:** Минимум 300 примеров (рекомендуется 500+).
**Баланс классов:**

| Тип сценария | Доля | Описание |
| --- | --- | --- |
| **Simple Chat** | 30% | Обычный диалог. Учит модель *не* вызывать команды зря. |
| **Search Intent** | 25% | Вопросы, требующие `CMD:SEARCH` или `CMD:GOOGLE`. |
| **Implicit Save** | 20% | Скрытые факты в речи юзера, требующие `CMD:SAVE`. |
| **RAG Response** | 25% | Симуляция ответа системы и реакции модели на него. |

### Структура примера (RAG Response)

Это критически важная часть. Модель должна видеть **контекст истории**.

```json
{
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "Где я работаю?"},
    {"role": "assistant", "content": "МЫСЛЬ: Вопрос про работу. Ищу.\nCMD:SEARCH | место работы"},
    {"role": "system", "content": "SYSTEM_OBSERVATION: - [2025-01-01] Работает в Яндекс (Tags: работа)"},
    {"role": "assistant", "content": "МЫСЛЬ: Факт найден. Отвечаю.\nВы работаете в компании Яндекс."}
  ]
}

```

## 5. Процесс Обучения (Training Pipeline)

### 5.1. Инструментарий

* **Платформа:** Google Colab (T4 GPU) или локальная машина с GPU (NVIDIA).
* **Библиотека:** `Unsloth` (для ускорения и экономии памяти).
* **Метод:** LoRA (Low-Rank Adaptation) — обучение адаптеров, а не всей модели.

### 5.2. Гиперпараметры (Рекомендуемые)

* `max_seq_length`: 2048 (достаточно для диалога).
* `load_in_4bit`: True (для экономии памяти).
* `lora_rank (r)`: 16 (баланс качества и скорости).
* `learning_rate`: 2e-4.
* `epochs`: 1-3 (или `max_steps`: ~100-200 для малого датасета).
* `dataset_format`: ChatML или Qwen-chat template.

---

## 6. Критерии Приемки (Acceptance Criteria)

Модель считается готовой, если проходит следующий **Test Script** (в Ollama):

1. **Тест "Болтун":**
* *Input:* "Привет, расскажи шутку."
* *Output:* Текст шутки. **Нет** команд `CMD:`. **Нет** фраз "Я ищу".


2. **Тест "Секретарь" (Save):**
* *Input:* "Я перешел на Линукс."
* *Output:* `МЫСЛЬ: ... CMD:SAVE | Перешел на Linux | технологии, софт` (Без лишнего текста).


3. **Тест "Поисковик" (Search):**
* *Input:* "На чем я пишу?"
* *Output:* `МЫСЛЬ: ... CMD:SEARCH | язык программирования`


4. **Тест "Аналитик" (Google):**
* *Input:* "Погода в Москве."
* *Output:* `МЫСЛЬ: ... CMD:GOOGLE | погода Москва сейчас`


5. **Тест "Чтец" (RAG):**
* *Input:* (Контекст с `SYSTEM_OBSERVATION: ... Пишет на Go`)
* *Output:* "Судя по записям, вы пишете на Go." (Без повтора даты, без тегов в тексте ответа).



---

## 7. Интеграция и Развертывание

1. Экспорт весов LoRA в формат **GGUF**.
2. Создание `Modelfile` для Ollama.
3. В `Modelfile` прописать `STOP` слова: `User:`, `Assistant:`, `SYSTEM_OBSERVATION:`.
4. Обновление конфига Go-бота (`config.yaml`).
5. **Упрощение кода:** Удаление сложных регулярок из `parser.go` (модель должна выдавать чистый формат сама).
